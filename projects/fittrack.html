FitTrack

A full-stack fitness and nutrition tracker built in React 19 — the data layer and charts are both hand-rolled, no libraries doing the heavy lifting.

Type
Personal project
Stack
React 19 · TypeScript · Node/Express · MongoDB
Repo
github.com/Simon-Mathew/fitness-app-calories ↗

The problem

A dashboard that logs workouts, food, and daily activity needs each section to load and update independently without the whole UI flickering or sections fighting over state. A slow API call in one panel shouldn't freeze the rest of the page, and an optimistic update in one section shouldn't leave the others stale.

How it's built

The data layer is four domain hooks — useWorkouts, useGoals, useActivity, useFoodLog — each managing its own state independently:

  • Optimistic updates on mutations so the UI responds instantly before the server confirms.
  • Race-condition prevention — stale responses from slow requests are ignored when a newer request has already resolved.
  • Per-section independent loading states — one slow request doesn't block or flicker the rest of the dashboard.

The API is 15+ RESTful endpoints on a Node/Express server. Daily activity uses a Mongoose upsert pattern for idempotent logging — posting the same day twice updates rather than duplicates. A global error handler returns specialised validation responses so the client can surface the right message without guessing.

A dual TypeScript interface system separates raw API shapes from UI display models. The network layer and the view models can evolve independently — changing a response schema doesn't require touching component code, and vice versa.

The visualisations — a calorie ring and a weekly bar chart — are hand-coded SVG. No charting library, no UI component library anywhere in the app.

Notable decisions

Building the hooks and SVG charts by hand was deliberate. It forces you to understand the underlying mechanics — state synchronisation, SVG geometry, the exact moment a re-render should fire — rather than accepting an abstraction that hides all of it. The resulting code is longer, but it's entirely readable.

The dual interface system is a pattern worth carrying forward. Conflating API shapes with UI models is a common source of subtle bugs as a project grows — changes to the backend ripple into component logic in unpredictable ways. Keeping them separate keeps that boundary explicit.
