Expense Tracker 2.0

A personal finance app running Angular 20 with Zone.js completely removed — all reactivity driven by signals and computed values.

Type
Personal project
Stack
Angular 20 · TypeScript · RxJS · Express 5 · MongoDB
Repo
github.com/Simon-Mathew/expense-tracker-2.0 ↗

The problem

A personal finance dashboard needs to track expenses, budgets, and per-category spending — and it needs to stay fast as data grows. The interesting challenge here wasn't the feature set; it was the question of how much of Angular's runtime machinery the app actually needed.

How it's built

The app runs zoneless Angular via provideZonelessChangeDetection(). Zone.js is gone entirely. State is driven by signal() and computed(), and components use OnPush change detection throughout. The runtime only re-renders what actually changed, not what might have changed.

The budget feature uses RxJS forkJoin to fire parallel HTTP requests — one per category — aggregate per-category spending against user-defined limits, and render animated progress cards when the responses arrive.

Angular SSR is wired in with provideClientHydration(withEventReplay()). Reusable components use content projection via named ng-content slots. Styling uses CSS design tokens rather than hardcoded values throughout.

The API is Express 5 with full CRUD, PATCH partial updates, and a Mongoose upsert pattern for budget limits that auto-seeds default categories on first run — no manual setup needed.

Notable decisions

Going zoneless is the headline technical decision. It's a current, advanced Angular pattern that most junior developers haven't touched yet. Removing Zone.js forces a clean signal-based mental model — you can't rely on the runtime to detect side effects; you have to declare your reactive graph explicitly. That discipline shows up in how the components are structured.

Using forkJoin for the budget requests rather than sequential calls cuts the wait time proportionally to the number of categories. With seven default categories, that's seven requests in parallel instead of seven in a chain.

The MongoDB upsert for budget limits means the API is idempotent on first run. A PUT to set a budget either creates or updates — callers don't need to know whether a record exists, and the database stays clean.
