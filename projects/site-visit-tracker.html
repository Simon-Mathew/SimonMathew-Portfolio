Site Visit Tracker

A field-visit logger with authentication and role-based access built from scratch — JWT, bcrypt, and IDOR-aware owner-scoped queries.

Why this project

Most tutorials reach for Auth0, Clerk, or Supabase the moment authentication comes up. This project skips all of that — the goal was to build JWT authentication from scratch to understand what a managed service is actually doing underneath. The GeoStamp project (a related client build) gave me the domain model; this one is where I dug into the auth layer properly.

How it's built

Location capture works via two paths — in-app camera capture using the browser Geolocation API, and photo upload with EXIF parsing via exifr and a Claude vision fallback for photos with no metadata.

Authentication is hand-rolled from first principles:

  • A User model with bcrypt password hashing (bcryptjs, configurable salt rounds).
  • Signed JWTs issued on login, verified on every protected route via Express middleware.
  • A requireRole guard that reads the role from the verified JWT payload — no database lookup on each request.
  • Owner-scoped queries throughout: every data-fetching route filters on the authenticated user's ID, with deliberate IDOR awareness (no route returns another user's records unless the caller is a manager).

Roles: a regular field user sees only their own visit records. A manager sees everyone's. The distinction is enforced at the query level, not just the UI level.

MongoDB is the single source of truth. Revisit reminders run on a daily cron. Excel export is generated on demand via SheetJS, same pattern as GeoStamp. Images are stored in Cloudinary.

Notable decisions

Choosing not to use a managed auth service was the point. The bcrypt work, the JWT signing, the middleware, the IDOR thinking — these are the things that get abstracted away when you reach for Auth0. Building them manually meant genuinely understanding what "secure by default" requires at each layer.

Enforcing access at the query level rather than the route level is the right call even with role middleware in place. Middleware checks "can this user class access this route?" — that's necessary but not sufficient. The query needs to also check "is this record owned by this specific user?" Otherwise a logged-in regular user could request /visits/someone-elses-id and get a response.
