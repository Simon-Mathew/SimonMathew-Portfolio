GeoStamp

A site-visit logger for Brothers Trading Company — location captured automatically, an AI vision model reading data off photos, one-click Excel export.

Type
Client project (BTC field staff)
Stack
Angular · Node/Express · MongoDB · Claude AI · Cloudinary · SheetJS
Deployed
Render (GitHub CI)

The problem

Brothers Trading Company's field employees move between sites all day. They needed a fast, low-friction way to record each visit location, timestamp, and any relevant details without typing out coordinates by hand or relying on a spreadsheet that lives on someone's laptop.

Typing coordinates is slow and error-prone. Taking a photo of a location marker or signage is instant, but getting the structured data out of that photo used to require manual entry anyway. The app needed to close that gap.

How it's built

Location capture works via two paths, both sitting behind a shared ExtractionResult TypeScript interface so the rest of the app doesn't care which path ran:

  • In-app capture : the browser Geolocation API fires at the moment the user captures a location. Photos taken through the browser don't embed GPS in EXIF, so the coordinate comes from the API call instead.
  • Photo upload : reads embedded GPS and timestamp from EXIF metadata using exifr. When a photo has no metadata (common with older cameras or screenshots), it's sent to a Claude vision model (claude-haiku-4-5) which reads any on-image stamp coordinates, address, timestamp and returns it as structured JSON.

A router with selectable modes controls which strategy runs: auto reaches for AI only when cheap metadata isn't available; compare runs both and shows them side by side so staff can cross-check. The modes are auto, exif, vision, and compare.

Recognised site data is written to MongoDB. Source images are stored in Cloudinary. Excel export lets staff and managers download their recorded sites as a clean .xlsx file generated from MongoDB via SheetJS.

Notable decisions

The shared ExtractionResult interface is the core design move. Geolocation, EXIF, and the vision model are interchangeable outputs the app doesn't branch on how data arrived, it just consumes the shape. A photo with no metadata degrades gracefully to AI extraction instead of failing.

Runtime AI calls are metered per token, so auto mode keeps cost low by calling the model only when it's genuinely needed when EXIF returns nothing useful.

Excel sits on top of the database rather than being the system of record. MongoDB is the single source of truth. The spreadsheet is a read-only snapshot, regenerated on demand, so there's no sync problem between the two.
